一、概述

1.1 什么是 opencode

opencode 是一个开源的 AI 编程代理（AI Coding Agent），运行在终端中，帮助开发者完成各类软件工程任务。它可以理解代码上下文，使用多种工具在项目中进行代码搜索、文件编辑、命令执行等操作，堪称你的"AI 结对编程伙伴"。

截至 2026 年 6 月，opencode 已经获得超过 160,000 个 GitHub Star，拥有 900 多位贡献者，每月服务超过 750 万 开发者。

1.2 核心特性

特性 说明
开源免费 MIT 协议，代码完全可见可审计
多模型支持 支持 Claude、GPT、Gemini 及 75+ 模型提供商
LSP 集成 自动加载语言服务器协议，提供精准的代码智能
多会话并行 同时启动多个代理会话，互不干扰
会话共享 生成链接分享会话给团队
多界面 终端 TUI、桌面 App、IDE 插件三种使用方式
隐私优先 不存储用户代码或上下文数据
企业级管控 支持 MDM 策略、托管配置、权限分级

1.3 适用场景

  • 代码搜索与导航：快速定位函数定义、查找所有引用、探索项目结构
  • 漏洞修复：定位 Bug、分析根因、实施修复
  • 功能开发：从需求分析到编码实现的全流程辅助
  • 代码重构：变量重命名、函数提取、跨文件重构
  • 技术调研：阅读理解陌生代码、探索依赖库实现
  • 文档撰写：生成 API 说明、技术文档、README
  • 自动化操作：安装依赖、配置 CI/CD、执行重复性变更
  • 代码审查：静态分析、安全审计、最佳实践检查

四、配置文件详解

4.1 配置文件位置

opencode 的配置遵循特定的优先级顺序（后加载的覆盖先加载的）：

优先级 配置源 路径 说明
1（最低） 远程配置 组织 .well-known/opencode 企业默认值
2 全局用户配置 ~/.config/opencode/opencode.json 个人偏好
3 自定义路径 $OPENCODE_CONFIG 环境变量 手动指定
4 项目配置 项目目录/opencode.json 项目特有设置
5 内联配置 OPENCODE_CONFIG_CONTENT 环境变量 运行时覆盖
6（最高） 托管配置 macOS: /Library/Application Support/ 管理员强制执行

配置文件的合并策略是合并而非替换：非冲突项全部保留，冲突项以后加载的为准。

五、AGENTS.md 规则系统

5.1 什么是 AGENTS.md

AGENTS.md 是 opencode 的规则文件，类似于 Cursor 的规则或 Claude Code 的 CLAUDE.md。它包含项目特定的指令，会在每次对话中被注入到 LLM 的上下文中。

5.2 文件优先级

opencode 在启动时会按以下顺序查找规则文件：

  1. 项目级：从当前目录向上遍历到 Git 根目录，查找 AGENTS.md（CLAUDE.md 作为备选）
  2. 全局级：~/.config/opencode/AGENTS.md
  3. 兼容级：~/.claude/CLAUDE.md（未禁用兼容模式时）

同一层级只取第一个找到的文件。

七、代理系统详解

7.1 代理类型

opencode 的代理分为两类：主代理（Primary）和子代理（Subagent）。

主代理：直接对话的对象，可通过 Tab 键切换。

子代理：在主代理的对话中按需调用，或通过 @ 手动指定。

7.2 内置主代理

Build 代理（默认）

  • 模式：主代理
  • 权限：所有工具全部开放
  • 用途：日常开发，从编码到测试的全流程操作

进入 Build 模式后，opencode 拥有完整能力：读取、编辑、搜索、执行命令等。

Plan 代理（规划模式）

  • 模式：主代理
  • 权限：edit 设为 ask（需确认）、bash 设为 ask
  • 用途：在不实际修改代码的前提下，分析需求、设计方案

典型工作流：

  1. 切换到 Plan 模式（按 Tab 键）
  2. 描述需求：“我需要添加用户登录功能，包括邮箱密码登录和 OAuth 登录”
  3. opencode 返回详细实施计划
  4. 反馈意见，迭代计划
  5. 切回 Build 模式（按 Tab 键）
  6. 要求实施：“按计划开始实施”

十一、高级功能

11.1 多会话并行

opencode 支持同时运行多个会话。你可以在一个会话中让代理分析代码结构，同时在另一个会话中让代理实施修改。

会话管理：

  • Tab：切换主代理
  • @agent：在子会话中启动子代理
  • 子会话结果会自动回到父会话上下文

11.2 Plan → Build 切换工作流

这是 opencode 最推荐的工作方式：

  1. Tab → Plan：进入规划模式
  2. 描述需求：详细描述你要实现的功能
  3. 评审计划：阅读 opencode 返回的计划，反馈修改意见
  4. Tab → Build：确认计划后切回构建模式
  5. 执行实施：说"按计划开始实施"

11.3 撤销与重做

opencode 使用快照（Snapshot）系统跟踪文件变更：

  • /undo：撤销上一步修改（可多次执行）
  • /redo：重做被撤销的修改

快照功能默认开启。可在 opencode.json 中通过 snapshot: false 关闭。

十二、最佳实践与技巧

12.1 编写高质量指令

  • 明确具体：不要只说"修 Bug"，而是说"用户登录后跳转到了 404 页面，检查 auth/login.ts 中的重定向逻辑"
  • 提供上下文：“这个函数接收用户对象，返回格式化后的地址字符串，参照 src/utils/address.ts 中的格式”
  • 引用文件：使用 @ 引用相关文件，帮助 opencode 理解上下文
  • 分步描述：复杂任务拆分为多个步骤，逐一说明
  • 举一反三：“像处理 /notes 路由那样处理 /settings 路由的认证逻辑”

12.2 如何与代理高效协作

  • 先 Plan 再 Build：复杂功能先规划方案，确认后再实施
  • 使用探索代理：在不熟悉的代码库中，先用 @explore 探索
  • 并行任务：多个独立任务可以同时进行
  • 及时验证：每次修改后运行测试和 lint
  • 善用 /undo：结果不满意及时回退

12.3 项目管理技巧

  • 维护 AGENTS.md：让它成为项目的"操作手册"
  • 自定义命令：常用操作配置为快捷命令
  • 自定义代理：团队专业领域创建专用代理
  • 会话共享：复杂问题解决后共享会话给团队

12.4 安全保障

  • .env 保护：opencode 默认拒绝读取 .env 文件
  • 权限分级：生产环境使用 "ask" 权限模式
  • 密钥管理：通过环境变量或独立文件管理密钥
  • 代码审查：在 Plan 模式下审查 AI 的建议后再执行
  • 企业策略：使用 Policies 控制权限范围

12.5 效率提升技巧

  • 分批读取：大文件分段读取，避免上下文浪费
  • 批量处理：使用 replaceAll 处理重复性修改
  • 任务列表：复杂任务使用 todowrite 跟踪进度
  • 拖拽图片：将 UI 设计图拖入终端供 AI 参考
  • 使用便宜模型：简单任务使用 small_model 节省成本

12.6 .ignore 文件

十五、常见问题与故障排除

15.1 安装问题

Q: 安装后找不到 opencode 命令

A: 确认安装目录在 PATH 中，或使用 npx opencode 临时运行。

Q: Windows 下终端显示异常

A: 建议使用 WSL2 运行，或使用 Windows Terminal + WezTerm 等现代终端。

15.2 配置问题

Q: 配置文件不生效

A: 检查 JSON 格式是否正确。配置按优先级合并，确认没有被更高级的配置覆盖。

Q: 某些工具被拒绝访问

A: 检查 permission 配置，确保所需工具未被设为 deny。

15.3 模型问题

Q: 模型响应缓慢

A: 检查网络连接，或考虑使用响应更快的模型（如 claude-haiku、gpt-4o-mini）。

Q: 模型经常超出上下文限制

A: 可以考虑配置 compaction 参数，或减少单次对话中的文件读取量。

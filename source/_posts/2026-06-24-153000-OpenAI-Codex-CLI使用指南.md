## 一、概述

### 1.1 什么是 OpenAI Codex CLI

OpenAI Codex CLI 是 OpenAI 推出的终端原生 AI 编码代理，能够在你的本地开发环境中自主规划、编写、编辑和测试代码。与基于浏览器的 AI 助手不同，Codex CLI 可以读取你的整个代码库、执行 Shell 命令并管理文件变更——它是一个真正的软件工程协作者，而非文本补全工具。

Codex CLI 最初于 2025 年 4 月以 Node.js 原型发布，2025 年 6 月用 Rust 重写了核心引擎（现代码库 95.6% 为 Rust），启动时间从 ~800ms 降至 <50ms，内存使用减少了 60-70%。截至 2026 年 4 月，Codex CLI 已拥有超过 300 万周活跃用户、75,000+ GitHub Star，2026 年 3 月 npm 下载量达 1453 万——同比增长 177 倍。

### 1.2 核心特性

| 特性 | 说明 |
|:----|:----|
| **Rust 核心** | 毫秒级启动，低内存占用，跨平台二进制分发 |
| **多模型支持** | GPT-5.5、GPT-5.4、GPT-5.3-codex 等，支持本地模型 |
| **OS 原生沙箱** | macOS Seatbelt、Linux Bubblewrap、Windows AppContainer |
| **灵活审批策略** | 从完全自动到每次操作都需确认 |
| **MCP 集成** | 支持 Model Context Protocol，连接第三方工具 |
| **Agent Skills** | 跨平台可复用技能系统 |
| **非交互模式** | 适用于 CI/CD 自动化脚本 |
| **图片输入** | 支持截图、设计稿等多模态输入 |
| **内置 Web 搜索** | 支持缓存和实时 Web 搜索 |
| **多代理并行** | 同时启动多个代理线程 |
| **AGENTS.md** | 项目级指令文件，类似 Claude Code 的 CLAUDE.md |

### 1.3 适用场景

- **功能开发**：从需求到编码实现的全流程辅助
- **漏洞修复**：定位 Bug、分析根因、实施修复
- **代码重构**：跨文件重构、变量重命名、函数提取
- **代码审查**：静态分析、变更审查、安全审计
- **技术调研**：理解陌生代码库、探索依赖实现
- **文档生成**：API 文档、CHANGELOG、README
- **CI/CD 自动化**：自动修复 lint 错误、运行测试、生成发布说明
- **UI 开发**：根据设计截图实现前端组件

## 二、安装指南

### 2.1 系统要求

| 要求 | 最低配置 |
|:----|:--------|
| macOS | 12+ (Monterey) |
| Linux | Ubuntu 20.04+ / Debian 10+ |
| Windows | 10+（原生 PowerShell 或 WSL2） |
| Node.js | 18+（npm 安装方式需要） |
| 内存 | 4 GB |
| 网络 | 可访问 api.openai.com |

### 2.2 安装方式

#### 方式一：npm 安装（全平台通用）

```bash
npm install -g @openai/codex
```

**注意**：包名是 `@openai/codex`，而不是 `codex`。未加 scope 的 `codex` 包是 2012 年的另一个完全不同的工具。

验证安装：

```bash
codex --version
```

#### 方式二：Homebrew 安装（macOS）

```bash
brew install --cask codex
```

#### 方式三：官方安装脚本（macOS/Linux）

```bash
curl -fsSL https://chatgpt.com/codex/install.sh | sh
```

无人值守安装：

```bash
curl -fsSL https://chatgpt.com/codex/install.sh | CODEX_NON_INTERACTIVE=1 sh
```

#### 方式四：Windows 安装（WSL2）

Windows 上推荐使用 WSL2 以获得最佳体验：

```bash
# 安装 WSL2（如未安装）
wsl --install

# 在 WSL2 中安装 Node.js
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/master/install.sh | bash
nvm install 22

# 安装 Codex CLI
npm install -g @openai/codex
```

### 2.3 升级

新版本定期发布，升级方式与安装相同：

```bash
# npm
npm update -g @openai/codex

# 安装脚本
curl -fsSL https://chatgpt.com/codex/install.sh | sh

# Homebrew
brew upgrade --cask codex
```

## 四、基础使用

### 4.1 启动交互式 TUI

```bash
codex
```

进入全屏终端 UI，Codex 会读取当前目录的代码库信息。启动后可以在输入框中键入自然语言指令。

### 4.2 启动时附带指令

```bash
codex "解释这个项目的架构"
codex "修复 auth 模块中的所有安全漏洞"
codex "为 user 模型添加 CRUD API 端点"
```

### 4.3 指定工作目录

```bash
codex -C ./packages/api "重构这个包"
codex --cd /path/to/project "分析代码质量"
```

### 4.4 恢复会话

Codex 在本地存储对话记录，可以随时恢复之前的会话：

```bash
codex resume            # 显示最近会话列表供选择
codex resume --last     # 直接恢复最近会话
codex resume --all      # 显示所有目录的会话
codex resume <SESSION_ID>  # 按 ID 恢复特定会话
```

非交互模式下也可恢复：

```bash
codex exec resume --last "继续修复你发现的问题"
codex exec resume 7f9f9a2e-1b3c-4c7a-9b0e-... "实施计划"
```

### 4.5 键盘快捷键

| 快捷键 | 功能 |
|:-------|:-----|
| `Ctrl+C` | 取消当前操作（按两次退出） |
| `Ctrl+D` | 退出 Codex（按两次强制退出） |
| `Ctrl+L` | 清除终端屏幕 |
| `Ctrl+G` | 在外部编辑器中编辑提示 |
| `Ctrl+R` | 搜索提示历史 |
| `Ctrl+O` | 复制最新输出 |
| `Enter` | 执行中时插入新指令 |
| `Tab` | 代理工作时排队后续提示 |
| `Esc + Esc` | 编辑上一条消息 |
| `Esc` | 关闭导航抽屉 |
| `Up/Down` | 浏览草稿历史 |
| `@` | 模糊文件搜索并附加到对话 |
| `!` | 执行本地 Shell 命令 |
| `Alt+,` / `Alt+.` | 降低/提高推理努力级别 |

### 4.6 图片输入

Codex CLI 支持多模态输入，可以直接粘贴或传入图片：

```bash
# 分析截图问题
codex -i error-screenshot.png "这个 UI 有问题，分析并修复"

# 根据设计图实现组件
codex -i figma-export.png "用 React 实现这个组件，完全匹配设计"

# 对比多个图片
codex -i before.png,after.png "比较这两张图的差异"
```

## 五、模型选择

### 5.1 可用模型

| 模型 | 用途 | 访问权限 |
|:----|:----|:--------|
| `gpt-5.5` | 复杂编码、计算机使用、研究 | ChatGPT 登录独占 |
| `gpt-5.4` | 旗舰级模型，强推理 + 工具调用 | 所有方案 |
| `gpt-5.4-mini` | 快速高效，适合子代理和快速任务 | 所有方案 |
| `gpt-5.3-codex` | 针对复杂软件工程优化的编码模型 | 所有方案 |
| `gpt-5.3-codex-spark` | 近乎即时的迭代，纯文本研究预览 | Pro 独占 |

Codex 还支持任何兼容 Chat Completions 或 Responses API 端点的模型。

### 5.2 切换模型

```bash
# 命令行指定
codex -m gpt-5.4-mini "快速修复这个 lint 错误"
codex --model gpt-5.3-codex "重构支付模块"

# 会话内切换
# 输入 /model 命令，然后选择模型
```

### 5.3 本地模型

Codex 支持使用本地开源模型（无需 API Key）：

```bash
# 使用 LM Studio
codex --oss --local-provider lmstudio

# 使用 Ollama
codex --oss --local-provider ollama
```

### 5.4 推理努力级别

控制模型的思考深度：

| 级别 | 适用场景 |
|:----|:--------|
| `minimal` | 简单查询 |
| `low` | 简单问题、快速修复 |
| `medium` | 默认，均衡模式 |
| `high` | 复杂调试、架构设计 |
| `xhigh` | 最难的问题 |

会话内使用 `/model` 命令调整，或用 `Alt+,` / `Alt+.` 快捷键实时切换。

## 六、配置文件 (config.toml)

### 6.1 配置层级

Codex 使用 TOML 配置文件，优先级从高到低：

1. CLI 命令行标志
2. Profile 配置 (`-p profile-name`)
3. 项目配置 (`.codex/config.toml`)
4. 用户配置 (`~/.codex/config.toml`)
5. 系统配置 (`/etc/codex/config.toml`)
6. 默认值

### 6.2 基本配置

```toml
# ～/.codex/config.toml

# 模型设置
model = "gpt-5.3-codex"
model_reasoning_effort = "high"

# 安全策略
sandbox_mode = "workspace-write"
approval_policy = "on-request"

# 个性化
personality = "pragmatic"

# Web 搜索
web_search = "cached"

# 性能
service_tier = "fast"

# 开发者指令（类似 AGENTS.md）
developer_instructions = "始终使用 TypeScript，优先使用函数式编程模式"

# 功能开关
[features]
multi_agent = true
shell_tool = true
```

### 6.3 配置字段详解

| 字段 | 可选值 | 默认值 | 说明 |
|:----|:-------|:-------|:-----|
| `model` | 模型字符串 | - | 默认模型 |
| `model_reasoning_effort` | minimal/low/medium/high/xhigh | - | 推理努力级别 |
| `approval_policy` | untrusted/on-request/never/reject | untrusted | 审批策略 |
| `sandbox_mode` | read-only/workspace-write/danger-full-access | read-only | 沙箱模式 |
| `web_search` | disabled/cached/live | cached | Web 搜索 |
| `personality` | friendly/pragmatic/none | none | 沟通风格 |
| `service_tier` | auto/fast | auto | 服务层级 |
| `project_doc_max_bytes` | 整数 | 32768 | AGENTS.md 大小上限 |

### 6.4 命名 Profile

可以为不同场景配置不同的 Profile：

```toml
[profiles.fast]
model = "gpt-5.4-mini"
model_reasoning_effort = "low"
service_tier = "fast"

[profiles.thorough]
model = "gpt-5.4"
model_reasoning_effort = "xhigh"

[profiles.ci]
approval_policy = "never"
sandbox_mode = "workspace-write"

[profiles.paranoid]
approval_policy = "untrusted"
sandbox_mode = "read-only"
```

使用 Profile：

```bash
codex -p fast "修复 README 中的拼写错误"
codex -p thorough "重新设计认证模块架构"
codex -p ci "运行测试并修复失败用例"
```

### 6.5 项目信任配置

```toml
[projects."/home/user/work/my-repo"]
trust_level = "trusted"
```

### 6.6 项目级配置 (`.codex/config.toml`)

可以放在项目根目录的 `.codex/config.toml` 中，仅对受信任项目生效：

```toml
# .codex/config.toml
model = "gpt-5.3-codex"
sandbox_mode = "workspace-write"
approval_policy = "on-request"
```

## 七、AGENTS.md 项目指令

### 7.1 什么是 AGENTS.md

AGENTS.md 是 Codex CLI 的项目级指令文件，类似 Claude Code 的 CLAUDE.md。它在每个会话开始时自动加载，告诉 Codex 你的项目规范、编码约定和构建命令。

### 7.2 加载顺序

Codex 按以下优先级加载指令文件：

1. `~/.codex/AGENTS.override.md`（全局覆盖）
2. `~/.codex/AGENTS.md`（全局默认）
3. Git 根目录 → 当前目录链查找：
   - `AGENTS.override.md`
   - `AGENTS.md`
   - 回退文件名

靠近工作目录的文件优先级更高。

### 7.3 示例

**全局 ~/.codex/AGENTS.md：**

```markdown
## 工作约定
- 修改源文件后始终运行 `pnpm test`
- 安装依赖时优先使用 `pnpm`
- 遵循 ESLint 配置，不内联禁用规则
```

**项目级 AGENTS.md：**

```markdown
# 我的项目

## 架构
- 前端: Next.js + TypeScript
- API: FastAPI (Python 3.12)
- 数据库: PostgreSQL via SQLAlchemy

## 构建命令
- 测试: `npm test`
- 构建: `npm run build`
- Lint: `npm run lint -- --fix`

## 编码规范
- Python 使用 snake_case，TypeScript 使用 camelCase
- 新端点需要对应的集成测试
- 永远不要直接提交到 main 分支
- 所有 API 端点必须有 OpenAPI 文档
```

**子系统覆盖**：`services/payments/AGENTS.override.md` 可以覆盖更目录级别的规则。

### 7.4 自动生成

在 Codex 会话中输入 `/init` 即可自动生成 AGENTS.md 脚手架。

### 7.5 回退文件名

通过 `config.toml` 配置备选文件名：

```toml
project_doc_fallback_filenames = ["TEAM_GUIDE.md", ".agents.md"]
```

## 八、审批策略与沙箱

### 8.1 审批策略

| 模式 | 行为 |
|:----|:-----|
| `untrusted` | 仅已知安全的读命令自动批准，其余都需确认 |
| `on-request` | 模型自行判断何时需要确认 |
| `never` | 从不询问，失败静默返回给模型 |
| `reject` | 自动拒绝特定类别的提示 |

### 8.2 沙箱模式

Codex 使用 OS 原生沙箱——macOS 上使用 Seatbelt，Linux 上使用 Bubblewrap，Windows 上使用受限令牌：

| 模式 | 文件访问 | 网络 | 适用场景 |
|:----|:---------|:-----|:---------|
| **read-only** | 任意读，不可写 | 禁用 | 安全探索、代码审查 |
| **workspace-write** | 项目目录内读写 | 禁用 | 日常编码（Git 项目默认） |
| **danger-full-access** | 全文件系统访问 | 启用 | 系统管理任务（谨慎使用） |

### 8.3 常用组合

```bash
# 安全探索（只读，每次询问）
codex -s read-only -a untrusted

# 日常编码（Git 项目默认行为）
codex -s workspace-write -a untrusted

# 全自动模式（workspace-write + on-request 的快捷方式）
codex --full-auto

# 无安全限制（危险！）
codex --yolo

# 添加额外的可写目录
codex --add-dir /path/to/shared/data "处理这些数据文件"
```

### 8.4 会话内调整

在交互式会话中，使用 `/permissions` 命令调整权限，或使用 `/plan` 进入计划模式（只读探索）。

## 十、会话内斜杠命令

### 10.1 会话管理

| 命令 | 功能 |
|:----|:-----|
| `/new` | 在当前会话中开始新对话 |
| `/clear` | 清除终端屏幕和对话 |
| `/resume` | 恢复已保存的对话 |
| `/fork` | 克隆当前对话到新线程 |
| `/compact` | 总结对话以释放 Token |
| `/side` | 快速提问，不加入主对话 |
| `/copy` | 复制最新输出到剪贴板 |
| `/status` | 显示当前模型、策略、Token 用量 |
| `/quit` / `/exit` | 退出 Codex CLI |
| `/diff` | 显示 Git diff（含未跟踪文件） |

### 10.2 配置与模型

| 命令 | 功能 |
|:----|:-----|

## 十一、非交互模式 (codex exec)

### 11.1 基本用法

`codex exec`（简写 `codex e`）用于脚本化和自动化场景，不启动 TUI：

```bash
codex exec "更新 CHANGELOG.md，基于最近一次 tag 以来的 git 日志"
codex e "运行测试并修复所有失败"   # 简写形式
```

### 11.2 JSON 输出

```bash
codex exec --json "列出所有 API 端点" | jq '.message'
codex exec --json "运行测试，总结失败信息" > test-results.json
```

### 11.3 输出到文件

```bash
codex exec -o summary.md "为本次变更生成发布摘要"
codex exec -o docs/api.md "从源代码生成 API 文档"
```

### 11.4 恢复会话

```bash
codex exec resume --last "现在添加错误处理"
codex exec resume <SESSION_ID> "继续实施计划"
```

### 11.5 自动化工作流

```bash
# 全自动模式用于 CI
codex exec --full-auto "修复所有 ESLint 错误，不改变逻辑"

# 使用快速模型降低成本
codex exec --full-auto --config model=gpt-5.4-mini "修复 lint 错误"

# 不持久化会话

# 在 Git 仓库外运行
codex exec --skip-git-repo-check "分析这个目录的结构"
```

## 十二、MCP 集成

### 12.1 什么是 MCP

Model Context Protocol (MCP) 是一个开放标准，让 AI 代理通过统一接口调用外部工具——浏览器、数据库、API、设计工具等。Codex CLI 支持作为 MCP 客户端（连接其他 MCP 服务器），也支持作为 MCP 服务器（被其他工具调用）。

截至 2026 年 4 月，已有超过 200 个社区 MCP 服务器，覆盖 Postgres、MySQL、Figma、GitHub、Slack、Cloudflare Workers 等。

### 12.2 配置 MCP 服务器

在 `config.toml` 中配置：

```toml
# STDIO 传输（本地进程）
[mcp_servers.github]
command = ["npx", "-y", "@modelcontextprotocol/server-github"]
enabled = true
startup_timeout_sec = 30

[mcp_servers.playwright]
command = ["npx", "@playwright/mcp@latest"]
env = { PLAYWRIGHT_BROWSERS_PATH = "0" }

# HTTP 传输（远程服务器）
[mcp_servers.remote-db]
url = "https://my-mcp-server.example.com"
```

### 12.3 使用 MCP 工具

```bash
# 添加 MCP 服务器
codex mcp add playwright npx "@playwright/mcp@latest" --extension

# 列出已配置的服务器
codex mcp list

# 登录 MCP 服务
codex mcp login my-server

# 会话内查看 MCP 工具
/mcp
/mcp verbose   # 详细诊断信息
```

### 12.4 将 Codex 作为 MCP 服务器

```bash
codex mcp-server
```

这使得其他支持 MCP 的 AI 工具（如 Claude Code、Cursor 等）可以调用 Codex CLI 的能力。

### 12.5 实用 MCP 场景

- **Playwright MCP**：让 Codex 能够与网页交互，抓取文档、测试 UI
- **GitHub MCP**：管理 Issue、PR、代码审查
- **Database MCP**：查询数据库结构后编写迁移脚本
- **Figma MCP**：读取设计规格后生成 UI 组件

## 十三、Agent Skills 系统

### 13.1 什么是 Skills

Agent Skills 是一个开放、跨平台的标准，用于扩展 AI 编码助手的功能。一个 Skill 是一个包含 Markdown 文件（和可选的脚本）的文件夹，代理在启动时读取元数据，仅在任务匹配时才加载完整指令。

### 13.2 Skill 目录

Codex 从以下位置扫描 Skills：

- `.agents/skills/` — 项目级别（可通过 Git 共享）
- `~/.codex/skills/` 或 `~/.agents/skills/` — 个人级别
- `/etc/codex/skills/` — 系统级别

### 13.3 使用 Skills

在对话中使用 `$skill-name` 调用 Skill：

```txt
回顾我的代码库使用的技术栈，确保所有依赖都是最新的
检查项目中是否有常见的安全漏洞
为我的 API 端点生成 OpenAPI 规范
```

### 13.4 创建 Skill

在 `~/.codex/skills/` 下创建文件夹，包含 `SKILL.md`：

```markdown
# 安全审查 Skill

## 描述
对代码库进行安全漏洞扫描

## 用法
当用户要求进行安全审查时使用此 skill

## 指令
1. 检查依赖中已知的 CVE
2. 检查硬编码的密钥/密码
3. 检查 SQL 注入风险
4. 检查 XSS 漏洞
```

## 十九、最佳实践

### 19.1 项目配置

1. **始终创建 AGENTS.md**：用 `/init` 生成，包含构建命令、测试命令和编码规范
2. **使用 `.codex/config.toml`**：为项目设置默认模型和安全策略
3. **善用 Profile**：为不同场景（快速修复、深度分析、CI）配置不同 Profile

### 19.2 安全建议

1. **不要使用 `--yolo`**：仅在隔离环境中使用
2. **审查所有自动生成的代码**：AI 可能产生安全漏洞
3. **使用 `workspace-write` 沙箱**：限制文件访问范围
4. **使用 `add-dir` 共享目录**：而不是使用 `danger-full-access`
5. **定期检查规则文件**：确保执行策略符合团队规范

### 19.3 效率技巧

1. **使用 `/compact` 释放 Token**：长对话后压缩上下文
2. **使用 `/side` 快速提问**：不打断主对话流程
3. **使用 `Tab` 排队指令**：代理工作时可以准备下一条指令
4. **使用图片输入调试**：截图配合修复指令效率极高
5. **善用 `codex review`**：在 PR 前自动审查代码

### 19.4 CI/CD 集成

1. **使用 API Key 而非 OAuth**：避免浏览器交互
2. **设置审批策略为 `never`**：非交互模式不需要确认
3. **使用 JSON 输出**：便于下游工具解析
4. **设置花销预算**：使用 `gpt-5.4-mini` 控制成本

## 二十、常见问题排查

### 20.1 安装问题

**Q: "codex" 命令找不到**
A: 确认 npm 全局安装路径在 PATH 中：
```bash
npm list -g @openai/codex
```

**Q: Windows 上沙箱不工作**
A: Windows 支持仍为实验性，推荐使用 WSL2。

**Q: 权限错误**
A: 尝试以管理员身份运行 PowerShell。

### 20.2 认证问题

**Q: 浏览器没有打开**
A: 检查默认浏览器设置，或使用 `codex login --device-auth`。

**Q: API Key 认证失败**
A: 确认 Key 有效且有配额，检查网络是否能访问 `api.openai.com`。

### 20.3 运行时问题

**Q: 模型响应慢**
A: 使用 `gpt-5.4-mini` 或降低推理努力级别，检查网络连接。

**Q: Token 用尽**
A: 使用 `/compact` 压缩上下文，或开始新会话。

**Q: 无法写入文件**
A: 检查沙箱模式，使用 `-s workspace-write` 或 `--add-dir` 添加可写目录。

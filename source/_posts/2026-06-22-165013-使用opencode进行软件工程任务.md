## 一、概述

### 1.1 什么是 opencode

opencode 是一个开源的 AI 编程代理（AI Coding Agent），运行在终端中，帮助开发者完成各类软件工程任务。它可以理解代码上下文，使用多种工具在项目中进行代码搜索、文件编辑、命令执行等操作，堪称你的"AI 结对编程伙伴"。

截至 2026 年 6 月，opencode 已经获得超过 **160,000** 个 GitHub Star，拥有 **900** 多位贡献者，每月服务超过 **750 万** 开发者。

### 1.2 核心特性

| 特性 | 说明 |
|:----|:----|
| **开源免费** | MIT 协议，代码完全可见可审计 |
| **多模型支持** | 支持 Claude、GPT、Gemini 及 75+ 模型提供商 |
| **LSP 集成** | 自动加载语言服务器协议，提供精准的代码智能 |
| **多会话并行** | 同时启动多个代理会话，互不干扰 |
| **会话共享** | 生成链接分享会话给团队 |
| **多界面** | 终端 TUI、桌面 App、IDE 插件三种使用方式 |
| **隐私优先** | 不存储用户代码或上下文数据 |
| **企业级管控** | 支持 MDM 策略、托管配置、权限分级 |

### 1.3 适用场景

- **代码搜索与导航**：快速定位函数定义、查找所有引用、探索项目结构
- **漏洞修复**：定位 Bug、分析根因、实施修复
- **功能开发**：从需求分析到编码实现的全流程辅助
- **代码重构**：变量重命名、函数提取、跨文件重构
- **技术调研**：阅读理解陌生代码、探索依赖库实现
- **文档撰写**：生成 API 说明、技术文档、README
- **自动化操作**：安装依赖、配置 CI/CD、执行重复性变更
- **代码审查**：静态分析、安全审计、最佳实践检查

## 二、安装指南

### 2.1 系统要求

- **操作系统**：macOS、Linux、Windows（推荐 WSL2）
- **终端**：推荐使用现代终端（WezTerm、Alacritty、Ghostty、Kitty 等）
- **Node.js**：通过 npm 安装时需要 Node.js 18+

### 2.2 安装方式

#### 方式一：官方安装脚本（推荐）

```bash
curl -fsSL https://opencode.ai/install | bash
```

此脚本会自动检测操作系统架构，下载对应二进制并配置 PATH。

#### 方式二：通过 npm 安装

```bash
# 使用 npm
npm install -g opencode-ai

# 使用 bun
bun install -g opencode-ai

# 使用 pnpm
pnpm install -g opencode-ai

# 使用 yarn
yarn global add opencode-ai
```

#### 方式三：通过 Homebrew（macOS / Linux）

```bash
brew install anomalyco/tap/opencode
```

#### 方式四：Windows 安装

```powershell
# 使用 Chocolatey
choco install opencode

# 使用 Scoop
scoop install opencode

# 使用 npm
npm install -g opencode-ai
```

> **WSL 推荐**：Windows 用户建议使用 WSL2 获得最佳体验，详见 [Windows WSL 文档](https://opencode.ai/docs/windows-wsl)。

#### 方式五：Docker

```bash
docker run -it --rm ghcr.io/anomalyco/opencode
```

### 2.3 验证安装

```bash
opencode --version
opencode --help
```

### 2.4 获取 API 密钥

opencode 需要通过 `opencode.json` 配置文件或 `/connect` 命令配置模型提供商。推荐方式：

**方式一：OpenCode Zen（零配置入门）**

在 TUI 中运行 `/connect`，选择 opencode，访问 [opencode.ai/auth](https://opencode.ai/auth) 注册并获取 API Key。

**方式二：自有 API Key**

支持 Anthropic Claude、OpenAI GPT、Google Gemini 等 75+ 模型提供商，将 API Key 配置在环境变量或 `opencode.json` 中。

## 三、快速上手

### 3.1 启动 opencode

进入你的项目目录，直接运行：

```bash
cd /your/project
opencode
```

首次启动会显示 TUI（终端用户界面），包含以下区域：

- **对话区**：显示 AI 的回复内容
- **输入区**：底部供你输入指令
- **状态栏**：显示当前使用的模型、代理等状态信息

### 3.2 初始化项目

启动后首次使用的推荐步骤是运行 `/init` 命令：

```
/init
```

opencode 会自动扫描项目结构，分析关键文件（`package.json`、`tsconfig.json`、`go.mod`、`Cargo.toml` 等），并在项目根目录生成一份 `AGENTS.md` 文件，其中包含：

- 项目的构建、测试命令
- 项目架构和目录结构说明
- 编码规范和约定
- 特殊配置和注意事项

> `AGENTS.md` 建议提交到 Git 仓库，方便团队成员共享。

### 3.3 第一条指令

初始化完成后，你可以直接提问：

```
这个项目是做什么的？请解释一下项目架构。
```

也可以请求执行具体任务：

```
帮我找到 src/utils.ts 中所有被废弃的函数，并添加 JSDoc 注释。
```

### 3.4 基础对话技巧

- **提及文件**：输入 `@` 后模糊搜索文件名，opencode 会自动读取文件内容
- **拖拽图片**：将图片拖入终端，opencode 会读取并理解图片内容
- **多轮交互**：可以持续对话，opencode 会记忆上下文

## 四、配置文件详解

### 4.1 配置文件位置

opencode 的配置遵循特定的优先级顺序（后加载的覆盖先加载的）：

| 优先级 | 配置源 | 路径 | 说明 |
|:------|:------|:----|:----|
| 1（最低） | 远程配置 | 组织 `.well-known/opencode` | 企业默认值 |
| 2 | 全局用户配置 | `~/.config/opencode/opencode.json` | 个人偏好 |
| 3 | 自定义路径 | `$OPENCODE_CONFIG` 环境变量 | 手动指定 |
| 4 | 项目配置 | `项目目录/opencode.json` | 项目特有设置 |
| 5 | 内联配置 | `OPENCODE_CONFIG_CONTENT` 环境变量 | 运行时覆盖 |
| 6（最高） | 托管配置 | macOS: `/Library/Application Support/` | 管理员强制执行 |

> 配置文件的合并策略是**合并而非替换**：非冲突项全部保留，冲突项以后加载的为准。

### 4.2 基础配置示例

```json
{
  "$schema": "https://opencode.ai/config.json",
  "model": "anthropic/claude-sonnet-4-5",
  "autoupdate": true,
  "server": {
    "port": 4096
  }
}
```

### 4.3 模型配置

```json
{
  "model": "anthropic/claude-sonnet-4-5",
  "small_model": "anthropic/claude-haiku-4-5",
  "provider": {
    "anthropic": {
      "options": {
        "apiKey": "{env:ANTHROPIC_API_KEY}",
        "timeout": 600000,
        "setCacheKey": true
      }
    }
  }
}
```

- `model`：主力模型，用于复杂推理和编码
- `small_model`：轻量模型，用于标题生成等简单任务（可选）
- `provider.options.timeout`：请求超时（毫秒）
- `provider.options.chunkTimeout`：流式响应的块超时

### 4.4 权限配置

默认所有工具都处于 `allow` 状态，你可以调整权限来增加安全性：

```json
{
  "permission": {
    "*": "ask",
    "bash": {
      "*": "ask",
      "git *": "allow",
      "npm *": "allow",
      "grep *": "allow"
    },
    "edit": "allow",
    "read": {
      "*.env*": "deny"
    }
  }
}
```

权限取值：

- `"allow"` — 自动执行，无需确认
- `"ask"` — 执行前询问用户意见
- `"deny"` — 阻止执行

> opencode 默认禁止读取 `.env` 文件，以保护敏感信息。

### 4.5 TUI 配置

TUI 专门的配置放在 `tui.json` 中：

```json
{
  "$schema": "https://opencode.ai/tui.json",
  "theme": "tokyonight",
  "scroll_speed": 3,
  "mouse": true,
  "attention": {
    "enabled": true,
    "notifications": true,
    "sound": true,
    "volume": 0.4
  },
  "diff_style": "auto"
}
```

支持多种主题：`tokyonight`、`catppuccin`、`dracula`、`nord`、`github-dark` 等。

### 4.6 Shell 配置

```json
{
  "shell": "pwsh"
}
```

opencode 会自动检测系统默认 Shell，也可以手动指定。Shell 的选择会影响命令执行的表现。

### 4.7 环境变量引用

配置文件中可以使用环境变量和文件引用：

```json
{
  "model": "{env:OPENCODE_MODEL}",
  "provider": {
    "anthropic": {
      "options": {
        "apiKey": "{file:~/.secrets/anthropic-key}"
      }
    }
  }
}
```

- `{env:VAR_NAME}`：读取环境变量
- `{file:path/to/file}`：读取文件内容

### 4.8 指令文件引用

```json
{
  "instructions": [
    "CONTRIBUTING.md",
    "docs/guidelines.md",
    ".cursor/rules/*.md"
  ]
}
```

opencode 会将这些文件的内容合并到系统提示中，让 AI 了解项目规范。

### 4.9 附件图片配置

```json
{
  "attachment": {
    "image": {
      "auto_resize": true,
      "max_width": 2000,
      "max_height": 2000,
      "max_base64_bytes": 5242880
    }
  }
}
```

当图片超出限制时，会自动缩放；若缩放后仍超出则拒绝发送。

## 五、AGENTS.md 规则系统

### 5.1 什么是 AGENTS.md

`AGENTS.md` 是 opencode 的规则文件，类似于 Cursor 的规则或 Claude Code 的 `CLAUDE.md`。它包含项目特定的指令，会在每次对话中被注入到 LLM 的上下文中。

### 5.2 文件优先级

opencode 在启动时会按以下顺序查找规则文件：

1. **项目级**：从当前目录向上遍历到 Git 根目录，查找 `AGENTS.md`（`CLAUDE.md` 作为备选）
2. **全局级**：`~/.config/opencode/AGENTS.md`
3. **兼容级**：`~/.claude/CLAUDE.md`（未禁用兼容模式时）

同一层级只取第一个找到的文件。

### 5.3 实战案例：前端项目

```markdown
# SST v3 单体仓库项目
这是一个 SST v3 单体仓库，使用 TypeScript，通过 bun workspaces 管理包。

## 项目结构
- `packages/` - 所有工作区包（functions、core、web 等）
- `infra/` - 按服务拆分的基础设施定义
- `sst.config.ts` - 主 SST 配置

## 代码标准
- 使用 TypeScript 严格模式
- 共享代码放在 `packages/core/` 中
- 函数代码放在 `packages/functions/` 中

## 构建与测试
- 构建：`bun run build`
- 测试：`bun run test`
- Lint：`bun run lint`
- 类型检查：`bun run typecheck`
```

### 5.4 实战案例：Python 后端项目

```markdown
# FastAPI 后端项目

## 编码规范
- 遵循 PEP 8 风格指南
- 使用 4 个空格缩进
- 类型注解必须完整
- 使用 `async def` 定义路由处理函数

## 项目结构
- `app/` - 应用主代码
- `app/api/` - API 路由定义
- `app/models/` - SQLAlchemy 模型
- `app/schemas/` - Pydantic 模式
- `tests/` - 测试用例

## 测试
- 运行测试：`pytest`
- 带覆盖率：`pytest --cov=app`
- 在提交前必须通过所有测试
```

### 5.5 /init 命令自动生成

执行 `/init` 命令后，opencode 会：

1. 扫描项目中的重要配置文件
2. 分析目录结构
3. 询问 1-2 个针对性问题
4. 生成或更新 `AGENTS.md`

如果已经存在 `AGENTS.md` 文件，`/init` 会增量更新而非覆盖。

### 5.6 让 AGENTS.md 引用外部文件

你可以在 `AGENTS.md` 中指引 opencode 读取其他文件：

```markdown
## 外部文件加载
当遇到文件引用时（如 @rules/general.md），使用 Read 工具在需要时按需加载。

## 开发指南
- TypeScript 代码风格：@docs/typescript-guidelines.md
- React 组件模式：@docs/react-patterns.md
- API 设计标准：@docs/api-standards.md
```

或者使用 `opencode.json` 中的 `instructions` 字段（更推荐）：

```json
{
  "instructions": [
    "docs/development-standards.md",
    "test/testing-guidelines.md",
    "packages/*/AGENTS.md"
  ]
}
```

## 六、核心工具详细说明

### 6.1 工具概览

opencode 内置了 12+ 种工具，覆盖软件工程的各个方面：

| 工具 | 功能 | 典型用途 |
|:----|:----|:--------|
| **read** | 读取文件内容 | 查看源代码、配置文件 |
| **write** | 创建或覆写文件 | 新建组件、配置文件 |
| **edit** | 精确字符串替换 | 修改函数、修复 Bug |
| **glob** | 按文件名模式搜索 | 查找特定类型文件 |
| **grep** | 按内容搜索 | 搜索代码模式 |
| **bash** | 执行 Shell 命令 | 安装依赖、运行测试 |
| **todowrite** | 管理任务列表 | 记录待办事项 |
| **webfetch** | 获取网页内容 | 查阅在线文档 |
| **websearch** | 搜索网络 | 查找最新信息 |
| **question** | 询问用户 | 澄清需求、获取偏好 |
| **skill** | 加载专业技能 | 前端设计、文档协作 |
| **lsp** | 语言服务器协议 | 代码跳转、引用查找 |

### 6.2 read — 文件读取

读取项目中的文件内容，支持指定行范围。

**参数说明**：

- `filePath`：文件路径（必需）
- `offset`：起始行号（可选，从 1 开始）
- `limit`：最大读取行数（可选，默认 2000）

**使用场景**：

```
读取 src/app.ts 的前 50 行查看导入语句
读取 config.json 验证配置是否正确
读取超长日志文件时，先查看开头，再按偏移量分段读取
```

### 6.3 edit — 精确编辑

基于**精确字符串匹配**的修改方式，而非行号。这意味着即使文件后续发生变化，也能正确匹配。

**编辑四要素**：

1. `filePath`：目标文件
2. `oldString`：被替换的原文
3. `newString`：替换后的新内容
4. `replaceAll`：是否替换所有匹配项

**使用示例**：

将 `calculateTotal` 函数改为使用 BigInt：

```
oldString: function calculateTotal(items) {
newString: function calculateTotal(items: BigInt[]) {
```

批量重命名：

```
oldString: getCwd
newString: getCurrentWorkingDirectory
replaceAll: true
```

> **⚠️ 注意事项**：
> - 如果 `oldString` 匹配不到内容，操作会失败
> - 如果 `oldString` 匹配到多个位置且 `replaceAll` 为 false，操作会失败（需要提供更多上下文）
> - 建议在 `oldString` 中包含足够多的周围代码以确保唯一匹配

### 6.4 write — 文件写入

创建新文件或完全覆写已有文件。

**参数说明**：

- `filePath`：目标文件路径
- `content`：文件内容

**使用场景**：

```
创建新的 React 组件文件
编写测试文件
生成配置文件（如 .eslintrc.json、tsconfig.json 等）
```

### 6.5 grep — 内容搜索

使用正则表达式在项目中搜索文件内容。底层使用 ripgrep，性能优秀。

**参数说明**：

- `pattern`：正则表达式模式
- `path`：搜索目录（可选，默认当前项目）
- `include`：文件类型过滤（可选）

**示例**：

```
搜索所有包含 "TODO" 的行：pattern: "TODO"
搜索 useEffect 调用：pattern: "useEffect\("  include: *.tsx
搜索类定义：pattern: "class \w+Service"
搜索中文注释：pattern: "[\u4e00-\u9fff]+"
```

> `grep` 默认遵循 `.gitignore` 规则，忽略被排除的文件。若要搜索忽略的文件，需创建 `.ignore` 文件。

### 6.6 glob — 文件搜索

使用 glob 模式匹配文件名，结果按修改时间排序。

**参数说明**：

- `pattern`：glob 模式
- `path`：搜索目录（可选）

**示例**：

```
查找所有 TypeScript 源文件：**/*.ts
查找所有 React 组件文件：src/**/*.tsx
查找所有测试文件：**/*.test.ts
查找所有 Markdown 文档：**/*.md
```

### 6.7 bash — 命令执行

在项目环境中执行 Shell 命令。

**支持的操作**：

- 安装依赖：`npm install` / `pip install` / `go mod tidy`
- 运行测试：`npm test` / `pytest` / `go test`
- 代码检查：`npm run lint` / `ruff check`
- 构建项目：`npm run build` / `make` / `cargo build`
- Git 操作：`git status` / `git diff` / `git log`

**输出处理**：

- 输出超过 2000 行时自动保存到文件
- 可通过 Read 工具（指定 offset/limit）分页读取输出文件
- 超长的命令输出会触发截断，但会保留完整内容到文件中

### 6.8 todowrite — 任务跟踪

管理和跟踪多步骤任务的进度。

**支持的状态**：

- `pending`：等待执行
- `in_progress`：正在执行
- `completed`：已完成
- `cancelled`：已取消

**使用场景**：

```
记录一个 Bug 修复的步骤：
1. [x] 定位 Bug 位置
2. [ ] 理解相关代码逻辑
3. [ ] 实施修复方案
4. [ ] 运行测试验证
```

### 6.9 webfetch — 网页获取

获取指定 URL 的内容，常用于查阅在线文档、API 参考等。

**支持格式**：

- 纯文本（text）
- Markdown（markdown）
- 原始 HTML（html）

### 6.10 websearch — 网络搜索

搜索网络获取最新信息。需要使用 OpenCode 提供商或设置 `OPENCODE_ENABLE_EXA` 环境变量。

**使用场景**：

```
搜索最新的 React 19 特性
查找某个 npm 包的用法示例
确认某个 API 的最新变更
```

> 不需要额外 API Key，直接连接到 Exa AI 的托管服务。

### 6.11 skill — 加载技能

加载预设的专业技能（SKILL.md），为当前任务注入领域知识。

**可用技能**：

| 技能名称 | 功能描述 |
|:---------|:--------|
| algorithmic-art | p5.js 生成艺术 |
| brand-guidelines | 品牌色彩和排版 |
| canvas-design | 视觉艺术设计（海报、PDF） |
| doc-coauthoring | 文档协作撰写 |
| docx | Word 文档处理 |
| frontend-design | 前端界面设计 |
| internal-comms | 内部通信撰写 |
| mcp-builder | MCP 服务器开发 |
| pdf | PDF 处理 |
| pptx | 演示文稿制作 |
| skill-creator | 创建新技能 |
| slack-gif-creator | Slack GIF 制作 |
| theme-factory | 主题定制 |

**使用方法**：

opencode 会自动判断是否需要加载某项技能。也可以在指令中明确要求：

```
使用 frontend-design 技能帮我设计一个登录页面
```

### 6.12 question — 用户提问

在任务执行过程中向用户提问，获取偏好或澄清需求。支持单选、多选、自定义输入。

**使用场景**：

```
- 询问用户选择哪种实现方案
- 确认是否要引入新的依赖
- 在某些操作前征求用户许可
```

### 6.13 lsp — 语言服务器协议（实验性）

需要设置环境变量 `OPENCODE_EXPERIMENTAL_LSP_TOOL=true` 启用。

**支持的操作**：

- `goToDefinition`：跳转到定义

## 七、代理系统详解

### 7.1 代理类型

opencode 的代理分为两类：**主代理**（Primary）和**子代理**（Subagent）。

**主代理**：直接对话的对象，可通过 Tab 键切换。

**子代理**：在主代理的对话中按需调用，或通过 `@` 手动指定。

### 7.2 内置主代理

#### Build 代理（默认）

- **模式**：主代理
- **权限**：所有工具全部开放
- **用途**：日常开发，从编码到测试的全流程操作

进入 Build 模式后，opencode 拥有完整能力：读取、编辑、搜索、执行命令等。

#### Plan 代理（规划模式）

- **模式**：主代理
- **权限**：`edit` 设为 `ask`（需确认）、`bash` 设为 `ask`
- **用途**：在不实际修改代码的前提下，分析需求、设计方案

**典型工作流**：

1. 切换到 Plan 模式（按 Tab 键）
2. 描述需求："我需要添加用户登录功能，包括邮箱密码登录和 OAuth 登录"
3. opencode 返回详细实施计划
4. 反馈意见，迭代计划
5. 切回 Build 模式（按 Tab 键）
6. 要求实施："按计划开始实施"

### 7.3 内置子代理

#### General — 通用代理

- **模式**：子代理
- **用途**：执行多步骤复杂任务、并行处理
- **权限**：完整工具权限（不含 todowrite）

**调用方式**：

```
@general 帮我重构 utils.ts 中的所有工具函数
```

#### Explore — 探索代理

- **模式**：子代理
- **权限**：只读（不能修改文件）
- **用途**：快速探索代码库、理解架构

**调用方式**：

```
@explore 找到项目中所有的 API 路由定义
```

#### Scout — 侦察代理

- **模式**：子代理
- **权限**：只读（不能修改工作区文件）
- **用途**：查阅外部依赖和文档

**调用方式**：

```
@scout 查看 express 库中 Router 的实现
```

### 7.4 隐藏系统代理

opencode 还内置了一些自动运行的系统代理（用户不可见）：

| 代理 | 用途 |
|:----|:----|
| compaction | 上下文过长时自动压缩 |
| title | 自动生成会话标题 |
| summary | 生成会话摘要 |

### 7.5 创建自定义代理

可以通过 `opencode.json` 或 Markdown 文件创建自定义代理。

#### 方式一：JSON 配置

```json
{
  "agent": {
    "code-reviewer": {
      "description": "审查代码质量与最佳实践",
      "mode": "subagent",

      "temperature": 0.1,
      "prompt": "{file:./prompts/code-review.txt}",
      "permission": {
        "edit": "deny",
        "bash": "deny"
      }
    }
  }
}
```

#### 方式二：Markdown 文件

在 `~/.config/opencode/agents/` 或 `.opencode/agents/` 下创建 Markdown 文件：

```markdown

  edit: deny
  bash: deny
---

你是一个代码审查专家，专注于以下方面：
- 代码质量和最佳实践

- 性能影响
- 安全隐患

```

```bash
opencode agent create
```

交互式命令会引导你完成代理的创建。

### 7.6 代理配置选项

| 选项 | 类型 | 默认值 | 说明 |
|:----|:----|:------|:----|
| `description` | string | — | 代理描述（必要） |
| `mode` | string | `all` | `primary`、`subagent`、`all` |
| `model` | string | 主模型 | 代理使用的模型 |
| `temperature` | number | 模型默认 | 0.0~1.0，越低越确定 |
| `steps` | number | 无限制 | 最大迭代步数 |
| `permission` | object | 全局权限 | 代理级别权限覆盖 |
| `prompt` | string | — | 自定义系统提示 |
| `color` | string | — | UI 颜色（hex 或主题色） |
| `hidden` | boolean | false | 是否在 @菜单中隐藏 |
| `top_p` | number | 模型默认 | 替代温度参数 |

`temperature` 经验值：

- **0.0~0.2**：高度确定性，适合代码分析和审查
- **0.3~0.5**：适度创造性，适合通用开发
- **0.6~1.0**：高创造性，适合头脑风暴和探索

### 7.7 代理间的导航

- **`[Leader]+↓`**：进入第一个子会话
- **`→`**：切换到下一个子会话
- **`←`**：切换到上一个子会话
- **`↑`**：返回父会话

## 九、MCP 服务器集成

### 9.1 什么是 MCP

MCP（Model Context Protocol，模型上下文协议）是一种允许 AI 代理与外部工具和服务集成的开放标准。通过 MCP，opencode 可以连接数据库、API、第三方服务等。

### 9.2 配置本地 MCP 服务器

```json
{
  "mcp": {
    "my-local-mcp": {
      "type": "local",
      "command": ["npx", "-y", "my-mcp-command"],
      "enabled": true,
      "environment": {
        "MY_ENV_VAR": "my_value"
      }
    }
  }
}
```

**参数说明**：

| 参数 | 类型 | 必需 | 说明 |
|:----|:----|:----|:----|
| `type` | string | 是 | 必须为 `"local"` |
| `command` | array | 是 | 命令及参数 |
| `cwd` | string | 否 | 工作目录 |
| `environment` | object | 否 | 环境变量 |
| `enabled` | boolean | 否 | 是否启用 |
| `timeout` | number | 否 | 超时（ms，默认 5000） |

### 9.3 配置远程 MCP 服务器

```json
{
  "mcp": {
    "sentry": {
      "type": "remote",
      "url": "https://mcp.sentry.dev/mcp",
      "oauth": {}
    },
    "context7": {
      "type": "remote",
      "url": "https://mcp.context7.com/mcp",
      "headers": {
        "CONTEXT7_API_KEY": "{env:CONTEXT7_API_KEY}"
      }
    },
    "gh_grep": {
      "type": "remote",
      "url": "https://mcp.grep.app"
    }
  }
}
```

**OAuth 认证**：

对于需要 OAuth 的远程 MCP 服务器（如 Sentry）：

```bash
opencode mcp auth sentry
```

此命令会打开浏览器完成 OAuth 流程。

**查看 MCP 状态**：

```bash
opencode mcp list           # 列出所有 MCP 服务器及认证状态
opencode mcp auth list      # 查看认证状态
opencode mcp logout sentry  # 登出
```

### 9.4 按代理控制 MCP

如果 MCP 服务器很多，可以全局禁用、只在特定代理中启用：

```json
{
  "mcp": {
    "my-mcp": {
      "type": "local",
      "command": ["npx", "my-mcp-command"],
      "enabled": true
    }
  },
  "permission": {
    "my-mcp_*": "deny"
  },
  "agent": {
    "my-agent": {
      "permission": {
        "my-mcp_*": "allow"
      }
    }
  }
}
```

### 9.5 常用 MCP 示例

**GitHub MCP**：管理 Issue、PR、代码审查
**Sentry MCP**：查询错误和性能数据
**Context7 MCP**：搜索技术文档
**Vercel Grep MCP**：在 GitHub 上搜索代码示例

## 十、实战工作流程

### 10.1 修复 Bug 的完整流程

**场景**：用户报告表单提交后页面崩溃。

**步骤**：

1. **收集信息**
   ```
   @explore 找到所有与表单提交相关的文件
   ```

2. **理解代码**
   ```
   读取 src/forms/submitHandler.ts，解释表单提交流程
   ```

3. **定位原因**
   ```
   在 submitHandler.ts 中发现第 45 行未做空值检查
   ```

4. **实施修复**
   ```
   编辑文件，在调用 API 前添加空值校验
   ```

5. **验证结果**
   ```
   运行 npm test 确认所有测试通过
   ```

### 10.2 添加新功能的完整流程

**场景**：为博客添加"阅读量统计"功能。

**阶段一：规划（Plan 模式）**

1. 切换到 Plan 模式（Tab 键）
2. 描述需求："我想为博客文章的每篇文章添加阅读量统计，数据存数据库，前端显示数字"
3. 查看 opencode 返回的实施计划
4. 确认方案，切回 Build 模式

**阶段二：实施**

1. **数据库迁移**
   ```
   创建数据库迁移文件，为 posts 表添加 views_count 字段
   ```

2. **后端 API**
   ```
   创建 POST /api/posts/:id/views 路由，增加阅读计数
   创建 GET /api/posts/:id 路由，返回包含阅读量的文章数据
   ```

3. **前端组件**
   ```
   在文章详情页添加阅读量显示组件
   调用 API 获取和更新阅读量
   ```

4. **验证**
   ```
   运行 npm run lint 和 npm run typecheck
   启动开发服务器测试功能
   ```

### 10.3 代码重构流程

**场景**：将项目中已弃用的 `getCwd` 函数全部替换为 `getCurrentWorkingDirectory`。

1. **搜索所有使用位置**
   ```
   grep pattern: "getCwd" include: *.ts
   ```

2. **使用 replaceAll 批量替换**
   ```
   编辑文件，将 getCwd 替换为 getCurrentWorkingDirectory
   ```

3. **创建新函数**（如果需要）
   ```
   在 utils/path.ts 中添加 getCurrentWorkingDirectory 函数
   ```

4. **运行测试确认**
   ```
   npm test
   ```

### 10.4 理解遗留代码

**场景**：你刚接手一个已有 5 年历史的 Python 项目。

1. **项目概览**
   ```
   解释这个项目的整体架构和核心模块
   ```

2. **数据流追踪**
   ```
   @explore 追踪用户注册的完整数据流，从 HTTP 请求到数据库写入
   ```

3. **关键业务逻辑**
   ```
   阅读 app/services/order_service.py，解释订单状态机的状态转换
   ```

4. **生成文档**
   ```
   根据 project 中的代码，为 API 模块生成 OpenAPI 规范文档
   ```

### 10.5 技术债务清理

**场景**：项目中散布着 `// TODO:` 和 `// FIXME:` 注释，需要系统性地处理。

1. **搜集所有待办项**
   ```
   grep pattern: "(TODO|FIXME|HACK|XXX):" include: *.ts
   ```

2. **分类评估**
   ```
   将所有 TODO 按模块分类，评估每个的优先级和影响范围
   ```

3. **逐步处理**
   ```
   从高优先级的 TODO 开始，逐一处理
   ```

## 十一、高级功能

### 11.1 多会话并行

opencode 支持同时运行多个会话。你可以在一个会话中让代理分析代码结构，同时在另一个会话中让代理实施修改。

**会话管理**：
- `Tab`：切换主代理
- `@agent`：在子会话中启动子代理
- 子会话结果会自动回到父会话上下文

### 11.2 Plan → Build 切换工作流

这是 opencode 最推荐的工作方式：

1. **Tab → Plan**：进入规划模式
2. **描述需求**：详细描述你要实现的功能
3. **评审计划**：阅读 opencode 返回的计划，反馈修改意见
4. **Tab → Build**：确认计划后切回构建模式
5. **执行实施**：说"按计划开始实施"

### 11.3 撤销与重做

opencode 使用快照（Snapshot）系统跟踪文件变更：

- `/undo`：撤销上一步修改（可多次执行）
- `/redo`：重做被撤销的修改

> 快照功能默认开启。可在 `opencode.json` 中通过 `snapshot: false` 关闭。

### 11.4 会话共享

使用 `/share` 命令生成当前会话的分享链接：

```
/share
```

链接会自动复制到剪贴板。可以分享给团队成员查看对话记录。

**分享控制**：

```json
{
  "share": "manual"    // 手动分享（默认）
  // "share": "auto"   // 自动分享所有新会话
  // "share": "disabled" // 禁用分享
}
```

### 11.5 LSP 集成

opencode 可以集成 LSP（语言服务器协议），提供精准的代码智能。在 `opencode.json` 中配置：

```json
{
  "lsp": true
}
```

或者手动配置特定语言的 LSP：

```json
{
  "lsp": {
    "typescript": {
      "disabled": true
    },
    "rust-analyzer": {
      "disabled": false
    }
  }
}
```

需要设置 `OPENCODE_EXPERIMENTAL_LSP_TOOL=true` 来启用 LSP 工具。

### 11.6 代码格式化

```json
{
  "formatter": true
}
```

禁用特定格式化器或添加自定义格式化器：

```json
{
  "formatter": {
    "prettier": {
      "disabled": true
    },
    "custom-prettier": {
      "command": ["npx", "prettier", "--write", "$FILE"],
      "extensions": [".js", ".ts", ".jsx", ".tsx"]
    }
  }
}
```

### 11.7 文件监控忽略

```json
{
  "watcher": {
    "ignore": ["node_modules/**", "dist/**", ".git/**"]
  }
}
```

### 11.8 上下文压缩

当对话上下文接近用尽时，opencode 会自动进行上下文压缩：

```json
{
  "compaction": {
    "auto": true,
    "prune": false,
    "reserved": 10000
  }
}
```

- `auto`：上下文满时自动压缩（默认开启）
- `prune`：删除旧的工具输出以节省 token
- `reserved`：保留的 token 缓冲

## 十二、最佳实践与技巧

### 12.1 编写高质量指令

- **明确具体**：不要只说"修 Bug"，而是说"用户登录后跳转到了 404 页面，检查 `auth/login.ts` 中的重定向逻辑"
- **提供上下文**："这个函数接收用户对象，返回格式化后的地址字符串，参照 `src/utils/address.ts` 中的格式"
- **引用文件**：使用 `@` 引用相关文件，帮助 opencode 理解上下文
- **分步描述**：复杂任务拆分为多个步骤，逐一说明
- **举一反三**："像处理 /notes 路由那样处理 /settings 路由的认证逻辑"

### 12.2 如何与代理高效协作

- **先 Plan 再 Build**：复杂功能先规划方案，确认后再实施
- **使用探索代理**：在不熟悉的代码库中，先用 `@explore` 探索
- **并行任务**：多个独立任务可以同时进行
- **及时验证**：每次修改后运行测试和 lint
- **善用 `/undo`**：结果不满意及时回退

### 12.3 项目管理技巧

- **维护 AGENTS.md**：让它成为项目的"操作手册"
- **自定义命令**：常用操作配置为快捷命令
- **自定义代理**：团队专业领域创建专用代理
- **会话共享**：复杂问题解决后共享会话给团队

### 12.4 安全保障

- **.env 保护**：opencode 默认拒绝读取 `.env` 文件
- **权限分级**：生产环境使用 `"ask"` 权限模式
- **密钥管理**：通过环境变量或独立文件管理密钥
- **代码审查**：在 Plan 模式下审查 AI 的建议后再执行
- **企业策略**：使用 `Policies` 控制权限范围

```json
{
  "experimental": {
    "policies": [
      {
        "effect": "deny",
        "action": "provider.use",
        "resource": "openai"
      }
    ]
  }
}
```

### 12.5 效率提升技巧

- **分批读取**：大文件分段读取，避免上下文浪费
- **批量处理**：使用 `replaceAll` 处理重复性修改
- **任务列表**：复杂任务使用 todowrite 跟踪进度
- **拖拽图片**：将 UI 设计图拖入终端供 AI 参考
- **使用便宜模型**：简单任务使用 `small_model` 节省成本

### 12.6 .ignore 文件

默认情况下 grep 和 glob 遵循 `.gitignore`。如果希望搜索被忽略的目录（如 `node_modules`），可以创建 `.ignore` 文件：

```
!node_modules/
!dist/
!build/
```

企业可以开发内部 MCP 服务器，对接内部服务（Jira、Confluence、内部 API 等），并通过远程配置分发给团队。

## 十五、常见问题与故障排除

### 15.1 安装问题

**Q: 安装后找不到 opencode 命令**

A: 确认安装目录在 PATH 中，或使用 npx opencode 临时运行。

**Q: Windows 下终端显示异常**

A: 建议使用 WSL2 运行，或使用 Windows Terminal + WezTerm 等现代终端。

### 15.2 配置问题

**Q: 配置文件不生效**

A: 检查 JSON 格式是否正确。配置按优先级合并，确认没有被更高级的配置覆盖。

**Q: 某些工具被拒绝访问**

A: 检查 `permission` 配置，确保所需工具未被设为 `deny`。

### 15.3 模型问题

**Q: 模型响应缓慢**

A: 检查网络连接，或考虑使用响应更快的模型（如 claude-haiku、gpt-4o-mini）。

**Q: 模型经常超出上下文限制**

A: 可以考虑配置 `compaction` 参数，或减少单次对话中的文件读取量。

### 15.4 调试

```bash
# 查看完整的解析后配置
opencode debug config

# 查看 MCP 服务器调试信息
opencode mcp debug my-server
```
